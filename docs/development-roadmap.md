# PyFlow 后续开发方向与计划

本文基于当前仓库代码实现整理，重点参考了 `pyflow.py`、`pyflow_app/`、`flow_config.json`、`node_templates.json`、`README.md`、`tests/` 和 CI 配置。

## 当前代码现状

### 已有能力

- 已具备桌面端可视化流程编排能力：节点拖拽、连线、缩放、网格、节点状态展示。
- 已具备多 Flow 管理：新建、重命名、删除、切换，并持久化当前激活 Flow。
- 已具备节点编辑能力：名称、图标、描述、工作目录、终端类型、跳过执行、失败后继续、多命令排序。
- 已具备执行能力：支持 `CMD`、`PowerShell`、`Bash`，支持条件连接和节点级日志输出。
- 已具备基本校验：循环依赖、不可达节点、空命令、无效工作目录、重复条件分支。
- 已具备基础工程化：GitHub Actions 做 lint、pytest 占位、PyInstaller 打包与 tag release。

### 当前主要问题

- `pyflow.py` 已完成首轮模块拆分，但主入口仍保留较多 UI 组装与窗口协调逻辑，可继续瘦身。
- 自动化测试已补上基础覆盖，但仍以核心逻辑和轻量 UI 装配测试为主，执行链路与更多边界场景还有扩展空间。
- 配置结构与实际功能存在脱节，JSON 里已有部分字段未在 UI 和执行流程中真正使用。
- 当前执行模型以串行为主，DAG 中独立分支不能并行执行。
- 保存体验还不完整，加载与保存路径的行为不够直观。
- 模板主要偏 PowerShell / 文件操作，对 macOS/Linux 的开箱体验不足。

## 建议的开发方向

### P0：先做结构治理

这是最优先事项，原因是后续所有功能都依赖可维护性。

当前状态：该阶段已完成主要部分，后续重点转向补齐 schema、兼容策略和进一步瘦身主窗口入口。

1. 拆分模块
   - [已完成] 拆出 `models`：节点、命令、连线、Flow 管理。
   - [已完成] 拆出 `persistence`：配置加载、保存。
   - [已完成] 拆出 `executor`：命令执行、状态流转、日志输出。
   - [已完成] 拆出 `ui`：画布、节点图元、编辑器、主窗口构建辅助。
   - [进行中] 继续收敛 `pyflow.py` 中的窗口协调和组装逻辑。

2. 稳定配置 schema
   - [部分完成] 已清理一批未落地字段，配置结构与当前实现更一致。
   - [未开始] 给配置增加版本号，便于未来迁移。
   - [未开始] 为配置读写增加兼容层，避免历史文件失效。

3. 建立最小测试集
   - [已完成] 已覆盖非 GUI 核心逻辑：拓扑排序、流程校验、控制器与部分窗口辅助逻辑。
   - [已完成] 已补充配置读写 round-trip 测试。
   - [部分完成] 已有执行控制器测试，执行路由和更多集成场景仍可继续补强。

## 推荐路线图

### 第一阶段：可维护性和安全迭代基础（1~2 周）

目标：让项目从“能跑的单文件原型”进入“可以持续加功能的工程”。

当前状态：本阶段大部分目标已落地。

- [已完成] 拆分 `pyflow.py`，已拆出模型、执行、持久化、工作区、主题和 UI 相关模块。
- [已完成] 新建 `tests/`，补上核心逻辑测试。
- [未开始] 给配置文件增加 schema 说明和版本字段。
- [部分完成] 已抽离保存/加载逻辑到工作区与持久化模块，路径体验仍可继续优化。

交付结果：

- [已达成] 核心逻辑可测试。
- [已达成] 主流程变更风险可控。
- [已达成] 后续功能开发成本明显下降。

### 第二阶段：补齐产品能力缺口（1~2 周）

目标：让 PyFlow 从“命令编排器”更接近“可复用工作流工具”。

- 为节点增加环境变量配置。
- 支持参数化输入，例如变量占位、流程级变量、节点级变量。
- 补齐 `flow_config.json` 中已经预留但未落地的字段。
- 增加执行摘要：开始时间、结束时间、耗时、结果统计。
- 在 UI 中增加更明显的日志入口和失败定位能力。

交付结果：

- Flow 可复用性更强。
- 调试体验更完整。
- 配置与实际功能保持一致。

### 第三阶段：增强易用性（1 周）

目标：提升第一次使用体验和跨平台实用性。

- 扩展模板库，不只提供 PowerShell 文件操作模板。
- 新增 Python、Git、Build、Test、Deploy 等常用模板。
- 按类别组织模板，并提供模板说明。
- 优化 README 和示例 Flow，让用户打开后能立即理解如何使用。

交付结果：

- 新用户更容易上手。
- macOS / Linux 用户的默认体验更好。
- 产品更接近可演示、可推广状态。

### 第四阶段：增强执行能力（2+ 周）

目标：从基础流程执行升级到更强的任务编排能力。

- 为独立分支引入可选并行执行。
- 增加取消、重试、跳过失败节点后的恢复策略。
- 增加执行历史视图，保留最近若干次运行记录。
- 为节点增加更多状态，例如等待中、执行中、成功、失败、已跳过。

交付结果：

- 更适合中等规模流程。
- 运行效率和可观测性更好。
- 更接近真正的 DAG 编排工具。

## 建议优先级

如果按投入产出比排序，建议优先做下面 6 件事：

1. 拆分 `pyflow.py` 模块结构。
2. 为 Flow 校验和执行核心补测试。
3. 落地环境变量和参数化能力。
4. 修正保存/加载路径行为。
5. 增强模板库和示例 Flow。
6. 增加执行摘要与历史记录。

## 可执行计划

### 近期计划（本月）

- [已完成] 完成模块拆分第一步：模型、配置、执行器独立。
- [已完成] 新增 10~20 个核心测试用例。
- [已完成] 支持 `requirements.txt` 对应的本地运行、打包脚本和文档说明。
- [未开始] 设计变量与环境变量的数据结构。

### 中期计划（1~2 个月）

- 完成变量系统和节点环境变量 UI。
- 提供更多跨平台模板。
- 增加执行历史、耗时统计、失败定位。
- 统一配置文件升级与兼容策略。

### 中长期计划（2~3 个月）

- 评估并实现并行执行。
- 将 UI 和执行引擎继续解耦。
- 为大型 Flow 提供更好的性能优化和交互反馈。
- 视需求考虑插件化节点或模板市场化方向。

## 结论

当前 PyFlow 已经具备一个桌面流程编排器原型应有的核心能力，下一阶段不建议继续把功能直接堆进单文件，而应优先做结构拆分、测试补齐和配置能力落地。这样可以在不失去当前可用性的前提下，把项目逐步推进到“可持续演进”的状态。
