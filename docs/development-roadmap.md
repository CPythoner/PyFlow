# PyFlow 后续开发方向与计划

本文基于当前仓库代码实现整理，重点参考了 `pyflow.py`、`flow_config.json`、`node_templates.json`、`README.md` 和 CI 配置。

## 当前代码现状

### 已有能力

- 已具备桌面端可视化流程编排能力：节点拖拽、连线、缩放、网格、节点状态展示。
- 已具备多 Flow 管理：新建、重命名、删除、切换，并持久化当前激活 Flow。
- 已具备节点编辑能力：名称、图标、描述、工作目录、终端类型、跳过执行、失败后继续、多命令排序。
- 已具备执行能力：支持 `CMD`、`PowerShell`、`Bash`，支持条件连接和节点级日志输出。
- 已具备基本校验：循环依赖、不可达节点、空命令、无效工作目录、重复条件分支。
- 已具备基础工程化：GitHub Actions 做 lint、pytest 占位、PyInstaller 打包与 tag release。

### 当前主要问题

- `pyflow.py` 体量过大，UI、数据模型、配置读写、执行逻辑全部耦合在单文件内。
- 自动化测试几乎为空，后续重构和加功能风险高。
- 配置结构与实际功能存在脱节，JSON 里已有部分字段未在 UI 和执行流程中真正使用。
- 当前执行模型以串行为主，DAG 中独立分支不能并行执行。
- 保存体验还不完整，加载与保存路径的行为不够直观。
- 模板主要偏 PowerShell / 文件操作，对 macOS/Linux 的开箱体验不足。

## 建议的开发方向

### P0：先做结构治理

这是最优先事项，原因是后续所有功能都依赖可维护性。

1. 拆分模块
   - 拆出 `models`：节点、命令、连线、Flow 管理。
   - 拆出 `persistence`：配置加载、保存、版本兼容。
   - 拆出 `executor`：命令执行、状态流转、日志输出。
   - 拆出 `ui`：画布、节点图元、编辑器、主窗口。

2. 稳定配置 schema
   - 明确保留哪些字段，移除未使用字段或补齐实现。
   - 给配置增加版本号，便于未来迁移。
   - 为配置读写增加兼容层，避免历史文件失效。

3. 建立最小测试集
   - 优先覆盖非 GUI 逻辑：拓扑排序、流程校验、条件分支、终端类型判断。
   - 为配置读写增加 round-trip 测试。
   - 为执行路由增加最小集成测试。

## 推荐路线图

### 第一阶段：可维护性和安全迭代基础（1~2 周）

目标：让项目从“能跑的单文件原型”进入“可以持续加功能的工程”。

- 拆分 `pyflow.py`，先拆模型层和执行层。
- 新建 `tests/`，补上核心逻辑测试。
- 给配置文件增加 schema 说明和版本字段。
- 统一保存/加载逻辑，支持记住当前文件路径。

交付结果：

- 核心逻辑可测试。
- 主流程变更风险可控。
- 后续功能开发成本明显下降。

### 第二阶段：补齐产品能力缺口（1~2 周）

目标：让 PyFlow 从“命令编排器”更接近“可复用工作流工具”。

- 为节点增加环境变量配置。
- 支持参数化输入，例如变量占位、流程级变量、节点级变量。
- 补齐 `flow_config.json` 中已经预留但未落地的字段。
- 增加执行摘要：开始时间、结束时间、耗时、结果统计。
- 在 UI 中增加更明显的日志入口和失败定位能力。

交付结果：

- Flow 可复用性更强。
- 调试体验更完整。
- 配置与实际功能保持一致。

### 第三阶段：增强易用性（1 周）

目标：提升第一次使用体验和跨平台实用性。

- 扩展模板库，不只提供 PowerShell 文件操作模板。
- 新增 Python、Git、Build、Test、Deploy 等常用模板。
- 按类别组织模板，并提供模板说明。
- 优化 README 和示例 Flow，让用户打开后能立即理解如何使用。

交付结果：

- 新用户更容易上手。
- macOS / Linux 用户的默认体验更好。
- 产品更接近可演示、可推广状态。

### 第四阶段：增强执行能力（2+ 周）

目标：从基础流程执行升级到更强的任务编排能力。

- 为独立分支引入可选并行执行。
- 增加取消、重试、跳过失败节点后的恢复策略。
- 增加执行历史视图，保留最近若干次运行记录。
- 为节点增加更多状态，例如等待中、执行中、成功、失败、已跳过。

交付结果：

- 更适合中等规模流程。
- 运行效率和可观测性更好。
- 更接近真正的 DAG 编排工具。

## 建议优先级

如果按投入产出比排序，建议优先做下面 6 件事：

1. 拆分 `pyflow.py` 模块结构。
2. 为 Flow 校验和执行核心补测试。
3. 落地环境变量和参数化能力。
4. 修正保存/加载路径行为。
5. 增强模板库和示例 Flow。
6. 增加执行摘要与历史记录。

## 可执行计划

### 近期计划（本月）

- 完成模块拆分第一步：模型、配置、执行器独立。
- 新增 10~20 个核心测试用例。
- 支持 `requirements.txt` 对应的本地运行和文档说明。
- 设计变量与环境变量的数据结构。

### 中期计划（1~2 个月）

- 完成变量系统和节点环境变量 UI。
- 提供更多跨平台模板。
- 增加执行历史、耗时统计、失败定位。
- 统一配置文件升级与兼容策略。

### 中长期计划（2~3 个月）

- 评估并实现并行执行。
- 将 UI 和执行引擎继续解耦。
- 为大型 Flow 提供更好的性能优化和交互反馈。
- 视需求考虑插件化节点或模板市场化方向。

## 结论

当前 PyFlow 已经具备一个桌面流程编排器原型应有的核心能力，下一阶段不建议继续把功能直接堆进单文件，而应优先做结构拆分、测试补齐和配置能力落地。这样可以在不失去当前可用性的前提下，把项目逐步推进到“可持续演进”的状态。
